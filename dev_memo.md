# Hyperbot 開発メモ

## 目的

Hyperbot は Hyperliquid 向けのグリッド Bot として開発する。ただし、最初からレバレッジ付きの実注文を出すのではなく、読み取り、検証、仮想売買、Testnet の順に安全性を確認しながら進める。

このリポジトリでは、Bot 本体だけでなく、戦略を決めるための補助ツールも単独で使えるようにする。特に、過去価格取得とバックテストは、グリッド範囲や資金配分をユーザーが判断するために独立した CLI と Python モジュールとして維持する。

## 基本方針

1. 実注文より先に価格取得を作る。
2. 実注文より先に過去価格取得とバックテストを作る。
3. 実注文より先にペーパートレードを作る。
4. 注文処理より先にリスク管理を作る。
5. Mainnet より先に Testnet で単発注文を確認する。
6. Testnet 単発注文が安定してから Testnet グリッドに進む。
7. レバレッジは最後に段階的に追加する。
8. 本番投入前に監視、通知、ログ、復旧処理を入れる。

最重要の原則は、Bot が利益を狙う前に、想定外の注文、二重発注、過大ポジション、証拠金不足、接続異常で止まれることを確認することである。

## 実装フェーズ

### Phase 1: 読み取り系

注文を出さず、Hyperliquid の読み取り API と WebSocket を扱う。

実装対象:

- 設定ファイル
- Hyperliquid SDK 導入
- Testnet 接続
- `allMids` 取得
- `l2Book` 取得
- WebSocket 接続
- WebSocket 再接続処理
- `candleSnapshot` による過去価格取得

現状では、以下を実装済み。

- `src/hyperbot/config.py`
- `src/hyperbot/market_data.py`
- `src/hyperbot/ws.py`
- `src/hyperbot/historical_prices.py`
- `scripts/run_testnet.py`
- `scripts/watch_testnet_ws.py`
- `scripts/fetch_history.py`

### Phase 2: 戦略検討とバックテスト

グリッド設定を決めるため、過去価格を使って仮想的に売買を検証する。

実装対象:

- グリッド上限と下限
- グリッド本数
- グリッド間隔
- 使用する金額
- 手数料率
- 過去の約定シミュレーション
- 実現損益、未実現損益、総損益、ROI の集計

現状では、`src/hyperbot/grid_backtest.py` と `scripts/run_grid_backtest.py` を実装済み。バックテストは OHLC ローソク足を使う近似であり、ローソク足内の価格順序は分からない。そのため、新規買いと利確売りを同じローソク足内で同時成立させない保守的な前提にしている。

### Phase 3: ペーパートレード

実際の価格を使い、注文はメモリ上の仮想注文として扱う。

確認すること:

- 買いが約定したら上に売りを置く
- 売りが約定したら下に買いを置く
- 同じ価格に重複注文しない
- レンジ外では新規注文を止める
- 含み損が想定どおり計算される

この段階では実注文は出さない。

### Phase 4: リスク管理

注文処理より先に `risk.py` を作る。

最低限必要な判定:

- 最大注文数制限
- 最大ポジションサイズ制限
- 最大損失制限
- 最大レバレッジ制限
- 証拠金率チェック
- 清算価格との距離チェック
- レンジ外停止
- 連続エラー停止
- kill switch

実注文を出す直前には、必ずリスク管理を最後の判定層として通す。

### Phase 5: Testnet 単発注文

グリッド Bot として動かす前に、Testnet で単発注文だけを確認する。

確認すること:

1. 現在価格を取得する。
2. 現在価格より十分離した小さい指値注文を出す。
3. 注文 ID を保存する。
4. 未約定注文一覧を取得する。
5. 注文をキャンセルする。
6. キャンセル済みを確認する。

この段階で、注文、キャンセル、未約定注文取得、約定検知、ポジション取得を個別に安定させる。

### Phase 6: 注文管理と永続化

Bot のローカル状態と取引所の状態を同期する。

管理対象:

- 注文 ID
- 価格
- 方向
- サイズ
- 状態
- 作成時刻
- 対応するグリッド番号

永続化は最低限 SQLite でよい。Bot 再起動時に未約定注文とポジションを Hyperliquid から取得し、ローカル DB と照合する。不明な注文がある場合は、新規注文を止めて手動確認に回す。

### Phase 7: Testnet グリッド

Testnet 上で小さいグリッドを動かす。

初期条件:

- 銘柄: BTC または ETH
- レバレッジ: 1x
- グリッド本数: 3 から 5 本
- 注文サイズ: 最小
- 稼働時間: 数時間から 1 日

確認すること:

- グリッド注文が並ぶ
- 約定後に反対注文が出る
- レンジ外注文がキャンセルされる
- 重複注文しない
- WebSocket 切断後に復旧する
- Bot 再起動後に状態復元できる
- 損切り条件で停止する
- kill switch で全注文キャンセルできる

### Phase 8: レバレッジ対応

Testnet で安定した後に、レバレッジを段階的に追加する。

順序:

1. 1x
2. 2x
3. 3x

確認すること:

- 注文サイズ計算が正しい
- 必要証拠金が正しい
- 清算価格を監視できる
- isolated 設定になっている
- ポジションが膨らみすぎない
- 含み損で新規注文が止まる

最初から高レバレッジを使わない。

### Phase 9: 運用準備

本番に近い検証では、サーバーで Testnet Bot を常時稼働させる。

必要なもの:

- systemd または Docker Compose
- ログローテーション
- Discord、Slack、Telegram などへの通知
- 秘密鍵の安全な管理
- プロセス自動復旧
- API エラー時の停止
- WebSocket 切断時の再接続

### Phase 10: Mainnet 小額稼働

Testnet で安定した後に Mainnet へ移行する。ただし、最初は極小サイズにする。

初期条件:

- レバレッジ: 1x
- グリッド本数: 2 から 3 本
- 注文サイズ: 最小
- 稼働時間: 数時間

確認すること:

- 実資金で注文できる
- 手数料込みで損益計算できる
- Funding を考慮できる
- 約定検知が遅れない
- 注文キャンセルが確実に動く

## 実行時ワークフロー

起動時:

1. config を読み込む。
2. 秘密鍵または API ウォレット設定を読み込む。
3. Hyperliquid 接続を確認する。
4. 対象銘柄の価格と板を取得する。
5. アカウント状態を取得する。
6. 未約定注文を取得する。
7. ローカル DB と同期する。
8. 異常があれば停止する。

通常稼働:

1. WebSocket で価格更新を受信する。
2. 約定イベントを検知する。
3. 約定した注文を DB 更新する。
4. 反対側のグリッド注文を作る。
5. `risk.py` で発注可否を判定する。
6. 問題がなければ注文を出す。
7. 注文 ID を DB に保存する。

定期処理:

1. 未約定注文を再取得する。
2. ローカル状態と照合する。
3. 範囲外注文をキャンセルする。
4. 証拠金率をチェックする。
5. 清算価格をチェックする。
6. ログを出力する。
7. 必要に応じて通知する。

異常時:

1. 新規注文を停止する。
2. 未約定注文をキャンセルする。
3. 必要ならポジション縮小を検討する。
4. 通知する。
5. 手動確認まで再開しない。

## 開発上の注意

- 実注文を追加する前に、リスク管理を必ず実装する。
- 取引所へ送った注文は、レスポンスだけで判断せず、未約定注文一覧を再取得して照合する。
- Bot 再起動時は、必ず取引所状態とローカル状態を同期する。
- 不明な注文や不整合がある場合は、新規注文を止める。
- 価格、数量、損益は `Decimal` を使う。
- バックテスト結果は参考値であり、実運用の損益を保証しない。
- Mainnet は Testnet で長時間安定してから、小額で始める。
